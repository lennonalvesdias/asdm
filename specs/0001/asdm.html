RFC — Documento de Arquitetura

ASDM

Agent Skills Distribution Manager

Plataforma corporativa para gestão centralizada, distribuição segura e metrificação de agents, commands e skills — multi-provider por design.

Problema Princípios Arquitetura Registry CLI Providers Segurança Métricas Perfis Estrutura Fluxos Roadmap

01 Problema

Hoje, cada provider de AI coding agent tem seu próprio formato de configuração. O resultado é fragmentação, duplicação e zero governança.

🔀

Fragmentação de formatos

OpenCode usa .opencode/ com OCX, Claude Code usa .claude/ com SKILL.md, Copilot usa .github/agents/ com instructions.md. Cada dev mantém 3+ configurações.

🔓

Zero governança

Qualquer dev pode alterar agents/skills localmente. Não há versionamento central, auditoria de mudanças ou garantia de integridade dos arquivos distribuídos.

📊

Sem visibilidade

Não sabemos quais perfis estão sendo usados, quais skills são mais efetivos, nem se os devs estão com versões atualizadas. Zero métricas de adoção.

02 Princípios de Design

📝

Write Once, Emit Many

Um único source-of-truth (ASDM Registry) que emite configurações nativas para cada provider. O dev não precisa saber o formato de cada um.

🔐

Integrity by Default

Cada arquivo distribuído carrega um SHA-256. A CLI valida na sincronização. Manipulação local é detectada e reportada.

🎯

Profile-First

Perfis (fullstack, data, mobile) são cidadãos de primeira classe. Cada perfil define seu conjunto de agents, skills e commands.

🧩

Composable

Perfis herdam de uma base e compõem layers. Um dev mobile pode extender o perfil base + mobile + iOS sem duplicação.

04 ASDM Registry

O Registry é o coração da plataforma. Ele é um serviço HTTP que serve o manifest de versões e os assets (arquivos de agents, skills, commands) assinados com checksums.

Manifest Schema

{
  "version": "2.1.0",
  "published_at": "2026-03-31T12:00:00Z",
  "profiles": {
    "fullstack-engineer": {
      "extends": ["base"],
      "description": "Perfil para devs fullstack",
      "agents": ["code-reviewer", "tdd-guide", "architect"],
      "skills": ["docx", "pdf", "react-best-practices", "api-design"],
      "commands": ["review", "test", "deploy-preview"],
      "providers": ["opencode", "claude-code", "copilot"]
    },
    "data-analytics": {
      "extends": ["base"],
      "agents": ["data-pipeline", "sql-optimizer", "viz-builder"],
      "skills": ["pandas", "sql", "dbt", "jupyter"],
      "commands": ["query", "pipeline", "notebook"],
      "providers": ["opencode", "claude-code", "copilot"]
    },
    "mobile": {
      "extends": ["base"],
      "agents": ["mobile-architect", "ui-reviewer"],
      "skills": ["react-native", "swift-ui", "kotlin-compose"],
      "commands": ["build-ios", "build-android", "e2e"],
      "providers": ["opencode", "claude-code", "copilot"]
    }
  },
  "assets": {
    "agents/code-reviewer.md": {
      "sha256": "a1b2c3d4...",
      "size": 4521,
      "updated_at": "2026-03-30T10:00:00Z"
    }
  }
}

API Endpoints

EndpointMétodoDescrição
/v1/manifestGETRetorna manifest completo com checksums
/v1/profiles/:nameGETRetorna definição de um perfil específico
/v1/assets/:pathGETDownload do arquivo com header X-SHA256
/v1/syncPOSTRecebe checksums locais, retorna diff de arquivos
/v1/telemetryPOSTRecebe beacon de métricas da CLI
/v1/verifyPOSTValida integridade de um conjunto de arquivos

Hosting flexível: O Registry pode rodar como Cloudflare Worker (zero-infra), como container Docker interno, ou até servido diretamente de um bucket S3 com um CDN na frente — desde que os checksums estejam no manifest.

05 ASDM CLI

A CLI é o ponto de contato do desenvolvedor. Deve ser extremamente simples — idealmente um único comando resolve 90% dos casos.

Comandos Principais

# Sincroniza tudo para o perfil do dev
npx asdm sync

# Sincroniza perfil específico
npx asdm sync --profile fullstack-engineer

# Verifica integridade dos arquivos locais
npx asdm verify

# Lista perfis disponíveis
npx asdm profiles

# Mostra diff entre local e registry
npx asdm status

# Inicializa projeto com perfil
npx asdm init --profile data-analytics

# Força re-download de tudo
npx asdm sync --force

# Sync apenas para um provider específico
npx asdm sync --provider opencode

Configuração Local

Arquivo .asdm.json na raiz do projeto ou ~/.config/asdm/config.json para global:

{
  "registry": "https://asdm.internal.company.com",
  "profile": "fullstack-engineer",
  "providers": ["opencode", "claude-code", "copilot"],
  "auto_verify": true,
  "telemetry": true,
  "sync_on_git_pull": true,
  "lockfile": ".asdm-lock.json"
}

Lockfile (.asdm-lock.json)

Gerado automaticamente após cada sync. Garante reprodutibilidade e permite detectar drift.

{
  "synced_at": "2026-03-31T14:22:00Z",
  "manifest_version": "2.1.0",
  "profile": "fullstack-engineer",
  "files": {
    ".opencode/agents/code-reviewer.md": {
      "sha256": "a1b2c3d4...",
      "source": "agents/code-reviewer.md"
    },
    ".claude/agents/code-reviewer.md": {
      "sha256": "e5f6a7b8...",
      "source": "agents/code-reviewer.md",
      "adapter": "claude-code"
    },
    ".github/agents/code-reviewer.agent.md": {
      "sha256": "c9d0e1f2...",
      "source": "agents/code-reviewer.md",
      "adapter": "copilot"
    }
  }
}

06 Emit Adapters (Multi-Provider)

Cada provider tem seu formato nativo. O ASDM mantém uma camada de adaptadores que converte o formato canônico para o formato esperado por cada ferramenta.

ProviderAgentsSkillsInstructionsConfig
OpenCode (OCX) .opencode/agents/*.md .opencode/skills/*/SKILL.md AGENTS.md .opencode/opencode.jsonc
Claude Code .claude/agents/*.md .claude/skills/*/SKILL.md CLAUDE.md .claude/settings.json
GitHub Copilot .github/agents/*.agent.md Via instructions .github/copilot-instructions.md YAML frontmatter

Formato Canônico ASDM

O source-of-truth é um formato intermediário que os adapters consomem:

# Arquivo canônico: agents/code-reviewer.asdm.md
---
name: code-reviewer
description: "Revisa PRs com foco em segurança, performance e clean code"
type: agent
providers:
  opencode:
    model: anthropic/claude-sonnet-4
    permissions: ["read", "write"]
  claude-code:
    model: claude-sonnet-4-20250514
  copilot:
    on: pull_request
    safe-outputs: ["*.md", "review-*"]
---

# Code Reviewer Agent

Você é um code reviewer sênior. Sua função é...

## Regras
- Sempre verifique vulnerabilidades OWASP Top 10
- Sugira testes para lógica complexa
- Nunca aprove código sem tratamento de erro adequado
...

Adapter Pipeline

  • 1. Parse Lê o arquivo .asdm.md, separa frontmatter YAML do corpo Markdown
  • 2. Transform Aplica regras específicas do provider (ex: adicionar frontmatter on: para Copilot, gerar opencode.jsonc entries para OCX)
  • 3. Emit Grava o arquivo no path esperado pelo provider, com formatação nativa
  • 4. Checksum Calcula SHA-256 do arquivo emitido e registra no lockfile

07 Segurança e Integridade

🔒

SHA-256 em Tudo

Cada arquivo no manifest possui hash SHA-256. Na sincronização, a CLI compara hashes antes de aceitar qualquer arquivo.

🚨

Detecção de Tampering

asdm verify compara hashes locais contra o lockfile. Qualquer divergência gera alerta e pode bloquear execução.

🪝

Git Hooks

Hook de pre-commit que roda asdm verify. Se houver tampering, o commit é bloqueado até o dev fazer asdm sync --force.

📡

Drift Reporting

A telemetria reporta quando um dev está com arquivos modificados localmente. Dashboard mostra compliance por equipe.

Fluxo de Verificação

# Em .husky/pre-commit ou .git/hooks/pre-commit
npx asdm verify --strict

# Exit codes:
#   0 = tudo íntegro
#   1 = arquivos modificados (lista quais)
#   2 = lockfile ausente (precisa sync)
#   3 = versão desatualizada (nova versão no registry)

Modelo de Permissões (Registry)

RolePodeNão Pode
Platform Admin Publicar versões, criar perfis, ver todas as métricas —
Tech Lead Propor mudanças (PR), ver métricas do time Publicar direto sem aprovação
Developer Sincronizar, verificar, ver métricas próprias Alterar registry ou perfis

Importante: O modelo copy-and-own do OCX é ótimo para flexibilidade individual, mas conflita com governança. O ASDM assume um modelo de managed distribution: arquivos emitidos são tratados como artefatos gerenciados. Se o dev precisa customizar, ele cria um overlay local que é separado do managed set.

08 Metrificação

A CLI envia beacons leves (opt-in corporativo) para o Metrics Collector. Isso permite visibilidade total de adoção sem invadir a privacidade do desenvolvedor.

Dados Coletados

profile_active manifest_version sync_timestamp sync_duration_ms providers_active files_synced_count integrity_violations drift_detected

Schema do Beacon

{
  "event": "sync_complete",
  "timestamp": "2026-03-31T14:22:00Z",
  "machine_id": "sha256(hostname+user)[0:12]",
  "profile": "fullstack-engineer",
  "manifest_version": "2.1.0",
  "providers": ["opencode", "claude-code"],
  "files_synced": 24,
  "integrity_ok": true,
  "drift_files": [],
  "sync_duration_ms": 1230
}

09 Sistema de Perfis

Perfis são a unidade principal de organização. Eles usam herança e composição para evitar duplicação.

Modelo de Herança

┌─────────────────────────────┐
│           base              │  ← Agents e skills comuns a todos
│  - code-reviewer            │
│  - security-scanner         │
│  - conventional-commits     │
├──────────┬──────────┬───────┤
│ fullstack│  data    │mobile │  ← Extends base
│ + api    │ + pandas │ + rn  │
│ + react  │ + sql    │ + ios │
│ + infra  │ + dbt    │ + and │
└──────────┴──────────┴───────┘
                │
         ┌──────┴──────┐
         │ mobile-ios  │  ← Extends mobile
         │ + swift-ui  │
         │ + xcode     │
         └─────────────┘

Resolução de Perfil

A resolução segue deep merge com precedência do mais específico:

final_config = deep_merge(
  profiles["base"],
  profiles["mobile"],
  profiles["mobile-ios"],     // mais específico ganha
  local_overlay               // overlay do dev (se existir)
)

Overlay Local: Devs podem ter um .asdm-overlay.json para adicionar skills pessoais que não fazem parte do perfil corporativo. Esses overlays são aditivos — não podem remover ou substituir managed assets.

11 Fluxos Operacionais

A. Dev Sincroniza (dia a dia)

  • npx asdm syncDev roda o comando no terminal
  • CLI lê .asdm.jsonIdentifica profile, registry URL, providers ativos
  • POST /v1/syncEnvia checksums locais ao registry
  • Registry calcula diffRetorna apenas arquivos novos ou modificados
  • Download + verifyBaixa assets, verifica SHA-256 de cada um
  • Emit adapters rodamConverte para formato nativo de cada provider
  • Lockfile atualizado.asdm-lock.json gravado com novos hashes
  • Beacon de telemetriaEnvia métricas de sync ao collector

B. Platform Team Publica Nova Versão

  • PR no source repoTech lead abre PR com mudanças em agents/skills
  • CI: validate.ymlValida schemas, checa frontmatter, roda linting
  • Code review + mergeAprovação por pelo menos 2 platform admins
  • CI: publish.ymlBuild manifest (com checksums), deploy ao registry
  • Devs detectam no próximo syncCLI avisa que há nova versão disponível

C. Detecção de Tampering

  • Dev edita arquivo managedEx: altera .opencode/agents/code-reviewer.md manualmente
  • Pre-commit hook: asdm verifyDetecta SHA-256 divergente
  • Commit bloqueadoMensagem clara: "arquivo X foi modificado fora do ASDM"
  • Dev escolheasdm sync --force para restaurar, ou move customização para overlay
  • Beacon reporta violationDashboard de compliance atualizado

12 Roadmap de Implementação

FaseEscopoDuração Est.
Fase 1 — Foundation Source repo + manifest build + CLI básica (sync, verify) + adapter OpenCode 3–4 semanas
Fase 2 — Multi-Provider Adapters Claude Code + Copilot, formato canônico .asdm.md, profiles com herança 2–3 semanas
Fase 3 — Integrity Git hooks, verify --strict, lockfile, detecção de tampering, overlay system 2 semanas
Fase 4 — Metrics Telemetry beacons, metrics store, dashboards de adoção e compliance 2–3 semanas
Fase 5 — Polish Auto-sync on git pull, notificações de nova versão, documentação, onboarding 2 semanas

Dica de quick start: A Fase 1 já entrega valor. Mesmo sem multi-provider, ter um npx asdm sync que distribui profiles versionados para OpenCode com verificação de integridade já resolve os maiores problemas de governança.
